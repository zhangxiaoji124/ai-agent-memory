# 方案与实验对比分析

## 1. 对比范围说明

本文分两类对比：

- **A. 已实测对比（同口径）**：仅使用本项目在同一数据与脚本下产出的结果（可信可复现）。
- **B. 方案层面对比（非同机实测）**：与主流向量检索方案做架构与能力对比，用于技术定位，不作为绝对性能结论。

数据来源：

- `logs/latency_budget_summary.csv`
- `logs/experiment_summary.csv`
- `logs/latency_budget_evaluation.md`

---

## 2. 你的方案定位

你的系统可视为：**HNSW 磁盘化 + 图感知缓存 + 拓扑预取 + 增量写入链路（MemTable/WAL/Compaction）**。

核心目标不是做“纯内存极限 ANN”，而是做 **Agent 记忆场景下、偏 I/O 受限条件的工程化检索系统**。

---

## 3. 与主流向量检索方案对比（架构/能力）

## 3.1 FAISS / hnswlib（库级）

- **优势**
  - 算法实现成熟，纯 ANN 检索性能强（尤其内存驻留场景）。
  - 集成成本低，快速验证 recall/latency。
- **相对你的不足（面向赛题目标）**
  - 对“磁盘 I/O 路径优化”支持不完整（尤其异步预取/缓存策略联动）。
  - 缺少你这种写链路（WAL + Compaction）的一体化工程闭环。
- **结论**
  - FAISS/hnswlib 是强基线；你的方案在“存储层优化 + 可持续写入”方向更贴赛题。

## 3.2 Milvus / Qdrant / Weaviate（向量数据库）

- **优势**
  - 工程能力完整：集群、高可用、过滤、管理面完善。
  - 生产可用性强，生态丰富。
- **相对你的不足（竞赛维度）**
  - 系统较重，针对单机 I/O 微优化不一定是其核心目标。
  - 在赛题要求的“底层 I/O 创新点可解释性”上，黑盒程度更高。
- **结论**
  - 向量数据库适合产品化；你的方案更适合“底层机制优化型赛题”展示。

## 3.3 DiskANN（研究/工程方向）

- **优势**
  - 明确面向 SSD 检索和 I/O 优化，学术与工程影响大。
  - 在大规模磁盘 ANN 上通常具备更成熟的高性能实现。
- **你当前相对差距**
  - 预取、队列调度、缓存策略等仍是第一版，尚未达到 DiskANN 级工程深度。
  - 缺少 Linux 实机 io_uring 对照数据来证明尾延迟与吞吐收益。
- **结论**
  - 你的路线方向与 DiskANN 接近，当前阶段可定位为“可运行的竞赛工程原型”，仍需性能打磨。

## 4. 你的实验结果对比（已实测）

实验口径：

- SIFT 子集：`base=100000`, `query=200`, `k=10`
- 固定：`m=16`, `ef_construction=200`, `rebuild=0`, `recompute_gt=1`
- 变量：`ef_search`
- 每个配置重复 3 次，取均值

### 4.1 核心结果（均值）

- `ef=48`: recall@10=`0.6915`, avg=`6.2258ms`, p95=`12.2466ms`
- `ef=128`: recall@10=`0.7560`, avg=`8.1782ms`, p95=`20.2546ms`
- `ef=192`: recall@10=`0.7630`, avg=`9.5936ms`, p95=`23.0759ms`
- `ef=320`: recall@10=`0.7715`, avg=`11.2430ms`, p95=`28.0074ms`

### 4.2 预算对比结论

- **时延预算 A（avg <= 10ms）最优**：`ef=192`
  - recall@10=`0.7630`
  - avg=`9.5936ms`
  - p95=`23.0759ms`
- **全局最高召回**：`ef=320`
  - recall@10=`0.7715`
  - 代价是 avg 超过 `10ms`

### 4.3 与你内部历史结果对比

结合此前粗粒度实验（`64/128/256/512`）与本轮细粒度网格，结论一致：

- `ef_search` 增大可稳定提升 recall，但时延同步上升。
- `ef=192` 是当前版本“在线可用”最优折中点。

- 你的系统在“磁盘 ANN + I/O 优化”的工程方向上已经形成闭环，且有可复现实验。

## 6. 补齐实验进展（已完成 + 待 Linux 实机）

### 6.1 已完成：同机基线实验

已完成 3 组模式、各 3 次重复（同机同数据同口径）：

- `full`：完整路径（缓存 + 预取 + 写链路框架）
- `baseline1`：关闭预取/缓存
- `baseline2`：仅 HNSW 磁盘检索（不启用写优化相关逻辑）

结果文件：

- 明细：`logs/compare_modes_raw.csv`
- 汇总：`logs/compare_modes_summary.csv`
- 报告：`logs/compare_modes_report.md`

关键结论：

- 三种模式 `recall@10` 均为 `0.763`（本组参数下召回一致）。
- `full` 的第 1 次出现冷启动离群（avg 明显偏高），去极值后与基线接近。
- 当前 Windows 环境下 `io_uring_active=0`，尚不能体现 io_uring 的平台收益。
