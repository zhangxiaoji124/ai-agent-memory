# 提升方案（基于 2026-05-27 实测结果 + 实现状态对照）

> 本文基于一轮 Linux 实机实测（SIFT 128 维、GIST 960 维）与一次设计-代码逐项核对得出。
> 关联：`docs/Agent长时记忆-实现状态对照.md`（已同步修订）、`Agent 长时记忆存储引擎设计(1).md`（设计原文）。

## 0. 这轮实测改变了什么（方案出发点）

| 之前的认知 | 实测后的事实 |
|---|---|
| 召回 0.76 偏低，疑似图质量问题 | **是 bug，已修复**：`HnswIndex::robust_prune` 遮挡不等式 alpha 放错边，过度剪枝到每节点 ~1 邻居。修正后 SIFT 100K=**1.000**、GIST 10K(960d)=**0.999**，**超过设计 85% 基线** |
| learned 策略普遍 ~2.5× QPS | 只在 SIFT 成立；**GIST 上 learned 反而慢 14%**（预取排序开销无 I/O 收益） |
| io_uring 是核心卖点 | 评测机 liburing 过老 → io_uring 走 **noop**；且 64GB RAM + ~800MB 数据**全驻留内存，没有磁盘等待** |
| — | **建图速度成了头号墙**：GIST 100K 建图烧 **86 分钟 CPU**（960 维标量距离） |

**核心矛盾**：这是个「磁盘 I/O 优化」赛题，但当前所有评测都是数据全进内存的「伪磁盘」场景——**I/O 优化的核心收益（预取/缓存/尾延迟）至今没有被证明过**。这比任何单点优化都重要。

附：另两个本轮修复的 bug —
- `eval_disk --recompute-gt 1` 跑 0 条 query（`nq` 用了 `gt_all.size()`，而重算 GT 时 GT 文件未加载、size=0）。
- `add_edge` 每条边都重跑 O(cap²) 的 `robust_prune`；改为「未满直接追加、满了才剪枝」（hnswlib/DiskANN 同款），建图提速。

## 1. 提升方案（按「核心 KPI + 可证明性」排序）

### P0 — 让它在赛题真实场景跑得起来、且能证明

**1. SIMD 距离（AVX2/FMA + 运行时派发）** —— 一切的前提
- 证据：GIST 建图小时级；`VectorStore::l2_sq_block`、`ExternalVectorStore::l2_sq`、`HnswIndex::l2_sq` 三处全是标量循环。
- 方案：抽出统一的 `l2_sq_f32(a, b, dim)`，用 `__builtin_cpu_supports("avx2")` 运行时派发 AVX2+FMA（8-wide）实现，标量兜底；ARM(鲲鹏) 走 NEON。
- 预期：建图小时 → 分钟级；查询距离吞吐 4–8×。**不做这个，1M / 500GB 场景根本跑不起来。**

**2. 构造「数据 > 内存」的评测配置** —— 否则 I/O 优化无法证明
- 现状：`ram_budget` 远大于数据集 → 全 mmap 驻留 → 预取/缓存策略毫无用武之地（GIST learned 变慢就是铁证）。
- 行动：用 `--ram-budget-gb` 把可用内存压到数据集的 **10–20%**（对齐设计三区预算）。这时缓存命中率、预取收益、p99 尾延迟才有意义，GoVector / PAIC / 预取整套机制才被「激活」。

**3. 固化召回基线 + commit 修复 + 同步文档**
- commit 两个 bug 修复（`robust_prune` 不等式 + eval 的 `nq`）与建图加速。
- 用统一脚本固化 SIFT/GIST recall 基线（`--recompute-gt 1`）。
- `docs/Agent长时记忆-实现状态对照.md` 已过期（θ/PAIC/图重排/RemapCom 标成「未实现」，实际已有骨架）——已随本轮修订。

### P1 — 把「名义实现」补成「最小真实版」（投入小、可解释性强）

**4. ML-When 预取门控（最小真实版）** —— 直接解决 GIST learned 变慢
- 现状：PAIC 已有 `useful_prefetch_demand_hits` / `wasted_prefetch_evictions` 计数，但**只统计不反馈**。
- 方案：用 wasted/useful 比值做在线节流——waste 高就自动降 fanout/depth。这是设计里 ML-When 的最小落地，且用实测问题（无效预取）反向验证。

**5. 图重排接进 build/compaction** —— 局部性是预取生效的前提
- 现状：BFS + Gorder 已实现（`runtime/graph_reorder.*`），但仅离线工具 `tools/reorder_index.cpp`，未接流程；无 Porder。
- 方案：建完索引自动 reorder 一次，配合按 offset 预取；输出局部性指标做对照。

### P2 — 结构性 / 诚实定位

**6. PQ/SQ 量化**（设计未写，但 DiskANN 核心）
- 双重收益：压缩码驻留内存让「数据 > 内存」评测成立；减少高维距离量。对 GIST 960 维收益最大。

**7. NodeBlock 瘦身**：去掉 2KB `neighbor_offsets`（可由 `node_offset(id)` 推算）→ 块减半、同容量缓存翻倍。

**8. 写优化三件套（RemapCom / NobLSM / NVTable）诚实降级**
- 现状：全是进程内模拟（无真实 `nvme_ioctl`、无 Ext4 内核协同）。
- 建议：**不硬接真实硬件**（工作量大、平台强绑定、风险高）。在答辩里诚实定位为「机制验证」，用一个写放大 / 尾延迟对照实验侧面证明思路。

## 2. 答辩风险（最重要）

1. **可证明性**：当前评测数据全进内存，I/O 优化核心收益从未被证明。P0#2 是必做项。
2. **命名 vs 实现**：RemapCom/NobLSM/PAIC/ISVM 有同名类但是模拟 / 固定启发式。主动区分「已实现 / 模拟 / 愿景」。
3. **状态文档过期**：已修订，避免把现状说错。
